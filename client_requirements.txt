Project 7 — Contact Management API with User Authentication & Authorization
Client Requirement
The company has successfully deployed the Contact Management REST API.
However, there is a major security issue.
Currently, anyone who knows the API URL can access, modify, or delete contact information.
To protect sensitive customer data, the company wants to introduce a secure authentication and authorization system.
Only registered users should be able to access the API, and different users should have different permissions based on their roles.
The application should implement secure authentication using JWT (JSON Web Tokens) and password hashing.

Feature 1 — User Registration
Endpoint
POST /auth/register
Request
{
    "full_name": "Aayush Thakkar",
    "username": "aayush",
    "email": "aayush@email.com",
    "password": "MyPassword123",
    "role": "Employee"
}
Requirements
Username must be unique.
Email must be unique.
Password should never be stored in plain text.
Hash the password before saving.

Design Constraints
Never store plain text passwords.
Use password hashing for every user.
Generate JWT tokens after successful login.
Protect all contact endpoints.
Keep authentication logic separate from business logic.
Use dependency injection to obtain the current authenticated user.
Validate user roles before executing protected operations.
Return proper HTTP status codes.
Keep modules focused on a single responsibility.

Mentor Rule
This project is your introduction to application security.
In previous projects, you focused on functionality.
Now, your mindset should become:
"Who is making this request?"
before asking:
"What operation should they be allowed to perform?"
Every API endpoint should first verify identity (authentication) and then verify permission (authorization). That's how production backend systems are designed.
